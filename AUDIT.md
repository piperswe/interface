# Interface — full project audit (round 2)

Scope: performance, software architecture, DRY, testability, correctness & bugs.
Single-tenant, Cloudflare-Access-fronted; security findings de-prioritised.

Findings are graded **P0** (data loss, hangs, hard breakage), **P1** (visible
bug, real perf cost, or pervasive design smell), **P2** (worth fixing when
nearby), **P3** (nice-to-have).

Line numbers reference the file at the top of each section unless noted.
This audit was produced after the round-1 fixes landed; round-1 history is
preserved in git. Where a finding overlaps an earlier note, the section
calls out whether it is a regression or a remaining gap.

---

## 0. Top findings (read first)

1. **`MetaPanel` reads token-usage fields the DO never writes.** P0, §2.4.
   `MetaPanel.svelte:17–28` reads `snapshot.usage?.promptTokens`,
   `completionTokens`, `promptTokensDetails.cachedTokens`,
   `completionTokensDetails.reasoningTokens`. The DO writes `Usage` from
   `LLM.ts:53–60` — `inputTokens`, `outputTokens`, `cacheReadInputTokens`,
   `thinkingTokens` — and persists that exact shape via
   `JSON.stringify(usage)` (`ConversationDurableObject.ts:1130`). Result:
   prompt-tokens, completion-tokens, cached-tokens, reasoning-tokens all
   render as `—` on every assistant message. `totalTokens` happens to
   match, so it's the only field that works. The `MetaSnapshot.usage` type
   in `types/conversation.ts:16–22` declares the wrong shape, so neither
   the DO writer nor the UI reader has been blocked by typing.
2. **`npm test` requires a manual `svelte-kit sync` first.** P0, §2.7.
   On a clean checkout (or a CI runner that hasn't run `npm run check`
   beforehand), `vitest run` fails 30/30 test files with
   `[TSCONFIG_ERROR] Failed to load tsconfig for 'test/setup.ts': Tsconfig
   not found`. `tsconfig.json` extends `./.svelte-kit/tsconfig.json`,
   which is generated by `svelte-kit sync`. After running it manually,
   333 tests pass cleanly. `package.json:13` should read
   `"test": "svelte-kit sync && vitest run"` (mirroring `check`).
3. **`routeLLM` regression: SDK clients no longer cached.** P1, §2.2.
   Round 1 fixed this, but the providers refactor reintroduced it.
   `route.ts:11–35` always uses the config-overload constructor
   (`new AnthropicLLM(provider.apiKey, …)` and `new OpenAILLM(config, …)`),
   so each call constructs a fresh `Anthropic` / `OpenAI` SDK client.
   The classes still accept a pre-built client (the unit tests use that
   path), but the production routing path doesn't. Connection pooling
   defeated for every chat turn, every compaction call, every title
   generation.
4. **`migrate.ts:migrateLegacyModelList` is dead code.** P1, §2.6.
   Exported but never imported anywhere outside its own file. Either it
   is finished work that should be deleted, or the call site that's
   supposed to trigger it (presumably the first `+layout.server.ts`
   load, or a one-shot at startup) was lost during the providers
   refactor. New deploys with a stale `model_list` setting will silently
   end up with no models configured.
5. **Provider stream is not cancelled on abort.** P1, §2.1.
   `#generate` calls `llm.chat({...})` (line 907) without `request.signal`.
   The `for await` loop breaks on abort, which calls the iterator's
   `return()` — that *eventually* cascades to the SDK's reader cancel —
   but the underlying HTTP fetch keeps the connection open until the
   provider stops. Anthropic/OpenAI SDK clients accept an `AbortSignal`
   (both adapters already forward `request.signal` at `AnthropicLLM.ts:63`
   and `OpenAILLM.ts:72`). The DO has no `AbortController` and never
   constructs one. Cost issue, plus aborted generations keep producing
   tokens server-side at the operator's expense.
6. **D1 read fan-out per generation is large.** P1, §2.1 / §3.
   `#generate` does a sequential `Promise.all` of 4 reads
   (`thinking_budget`, `system_prompt`, `user_bio`, `listAllModels`),
   then `getResolvedModel` (which fetches provider + model = 2 more
   round trips), then `#buildToolRegistry` triggers `listSubAgents` +
   `listAllModels` (a second time) + `listAllGlobalModelIds` (a third
   time, because it does its own `listAllModels`), plus `listMcpServers`.
   That's ~10 D1 round trips per chat turn, mostly cache-able for the
   conversation lifetime.
7. **Toaster wired for nothing.** P1, §2.4.
   `lib/components/Toaster.svelte` is a self-contained toast UI; its
   docstring at `lib/toasts.ts:3` says it is "mounted in the root
   layout" — it isn't. `pushToast` is exported and never called. Settings
   forms `redirect(303, '/settings')` for save feedback so the user gets
   no visible "saved" confirmation. The wiring was either dropped or
   never landed.
8. **`messages.parts` and `messages.parts_html` store the same JSON twice.**
   P2, §4.1. Round 1 added `parts_html` to cache server-rendered HTML,
   but it stores a complete `MessagePart[]` JSON with `textHtml` enriched
   onto text/thinking parts — i.e. a full duplicate of `parts` with
   extra fields. The reader at `ConversationDurableObject.ts:1310–1313`
   prefers `parts_html` and falls back to `parts`. Storage cost ~2× per
   completed assistant row, plus two `JSON.stringify` calls on the
   write side. Easier alternatives: one column with HTML embedded, or a
   separate `parts_html_overrides: Record<index, html>`.
9. **DO loop drops abort signal forwarding to sub-agent and sandbox tools.**
   P2, §2.3. `agent.ts:150` plumbs `ctx.signal` into the inner LLM call,
   and `sandbox.ts` forwards `ctx.signal` into `execStream`. But the DO
   passes `ctx.signal` only as `undefined` (never sets it in the
   `registry.execute(...)` call at line 988–1002). So the carefully
   plumbed plumbing has no source. Wiring the DO's own `AbortController`
   into both `chat()` and the tool-execution context closes finding 5
   and this one in one move.
10. **Sub-agent's allowed-tools enforcement is split-brain.** P2, §2.3.
    `agent.ts` filters `innerTools` by `allowed` at line 122–125 (so the
    LLM never sees the disallowed tool descriptors) AND checks `allowed`
    again at line 181–193 when the model nonetheless emits a tool call.
    The latter branch can only fire if the model hallucinates a tool
    name not in the schema. If you trust schema enforcement, remove the
    runtime check; if not, drop the upfront filter and rely on runtime.
    Belt-and-braces is fine, but document it.

## 1. Bugs and correctness

### 1.1 Token usage shape — visible UI bug (**P0**)

**File:** `src/lib/components/MetaPanel.svelte:17–28`,
`src/lib/types/conversation.ts:16–22`,
`src/lib/server/durable_objects/ConversationDurableObject.ts:1130`,
`src/lib/server/llm/LLM.ts:53–60`.

Two separate type declarations describe `usage` and they don't agree.
`Usage` (LLM.ts) uses `inputTokens` / `outputTokens` /
`cacheReadInputTokens` / `cacheCreationInputTokens` / `thinkingTokens`.
`MetaSnapshot.usage` (types/conversation.ts) uses `promptTokens` /
`completionTokens` / `promptTokensDetails.cachedTokens` /
`completionTokensDetails.reasoningTokens`.

The DO writes the LLM-shape blob (`JSON.stringify(usage)`) into
`usage_json` and reads it back into a `MetaSnapshot`-shaped variable.
`MetaPanel.svelte` reads `MetaSnapshot.usage`. The renderer reads
fields that don't exist in the persisted JSON. End-user visible:
prompt-tokens, completion-tokens, cached-tokens, reasoning-tokens
always show `—`.

The "fresh re-count" path in `compactHistory` covers this defensively
(`ConversationDurableObject.ts:813`: `lastUsage.inputTokens ?? lastUsage.promptTokens`),
which is presumably how this slipped through — the fallback works.

**Fix:** make `MetaSnapshot.usage = Usage` (drop the OpenRouter-legacy
shape entirely). Update `MetaPanel.svelte` to read the new field names.
Add a snapshot test that asserts the rendered `<dd>` content for a
known usage payload so this doesn't regress.

### 1.2 Test suite broken on cold checkout (**P0**)

**File:** `package.json:13`, `tsconfig.json:2`.

`tsconfig.json` extends `./.svelte-kit/tsconfig.json`, which is
generated by `svelte-kit sync`. `package.json`'s `"test"` script is
`vitest run` with no sync prefix. On any environment that hasn't run
`npm run check` (or `dev`/`build`), every test file fails to transform:
`[TSCONFIG_ERROR] Failed to load tsconfig for 'test/setup.ts': Tsconfig
not found`. After `npx svelte-kit sync`, all 333 tests pass.

CI in particular needs `svelte-kit sync` before tests, or `test:` needs
to run it itself. Match the `check`/`typecheck` pattern:
`"test": "svelte-kit sync && vitest run"`.

Side note: the test run produces hundreds of `WebSocket peer disconnected`
and `Network connection lost` messages on stderr. They come from
incomplete teardown of `subscribe()` streams in
`ConversationDurableObject.test.ts` and don't fail the run, but they
overwhelm the actual test output. Worth fixing the readers in the
end-to-end tests to drain (or `cancel()`) the streams cleanly.

### 1.3 Provider stream not cancelled on abort (**P1**)

**File:** `src/lib/server/durable_objects/ConversationDurableObject.ts:907`.

`for await (const ev of llm.chat({ messages, systemPrompt, ...tools, ...thinking, ...reasoning }))`
constructs the request object literally. There is no `signal` field.
`abortGeneration` clears `#inProgress` and broadcasts a refresh; the
loop's per-event guard at line 914 then `break`s out. JS spec says the
generator's `return()` is then awaited — so the SDK's stream is
cancelled — but only after the next chunk arrives. In the meantime the
provider is producing tokens you've already given up on.

The adapter side is ready: `AnthropicLLM.ts:63` and `OpenAILLM.ts:72`
both forward `request.signal`. `agent.ts:150` plumbs `ctx.signal` into
sub-agent calls. `sandbox.ts:118` plumbs `ctx.signal` into `execStream`.
**None of these has a source** — the DO never constructs an
`AbortController` to feed any of them.

**Fix:** at the top of `#generate`, build
`const ac = new AbortController();` Stash on `this.#inProgress` so
`abortGeneration` can call `ac.abort('user')`. Pass `ac.signal` into
`llm.chat({ ..., signal: ac.signal })` and into `registry.execute({
..., signal: ac.signal })` (line 988). Closes findings §0.5 and §0.9.

### 1.4 `routeLLM` regression: SDK client per call (**P1**)

**File:** `src/lib/server/llm/route.ts:11–35`.

```ts
case 'anthropic': {
  if (!provider.apiKey) throw new Error(...);
  return new AnthropicLLM(provider.apiKey, model.id, provider.id);
}
case 'openai_compatible': {
  if (!provider.apiKey) throw new Error(...);
  const config: OpenAILLMConfig = { baseURL, apiKey, extraHeaders: {...} };
  return new OpenAILLM(config, model.id, provider.id);
}
```

Both branches go through the constructor that internally
`new Anthropic({...})` / `new OpenAI({...})`. Connection pool inside the
SDK is recreated per `routeLLM` call. `#generate` calls `routeLLM` once
per turn; `compactHistory` calls it once per compaction; `#writeTitle`
calls it once per title-gen. That's at least three reconstructions per
heavy turn, all hitting the same provider.

The constructor overloads support cached clients; tests already use the
"pass me a pre-built client" path. Fix: keep a module-scope
`Map<provider.id, { client, fingerprint }>` keyed by provider id, with
the value invalidated when the api key changes. The fingerprint is a
hash of `apiKey + endpoint + extraHeaders` so a settings save naturally
cycles the cached client.

Round 1 fixed this; the providers-table refactor reintroduced it.

### 1.5 `compactHistory` is not exercisable through the DO's test seam (**P1**)

**File:** `src/lib/server/llm/context.ts:62–157`,
`src/lib/server/durable_objects/ConversationDurableObject.ts:817`.

`compactHistory` accepts `deps.llm` for injection, which is great —
unit tests use it. The DO's `#generate` calls `compactHistory(messages,
model, this.env, usageForCompaction)` *without* the deps argument. So
when a DO test (`ConversationDurableObject.test.ts`) wants to assert
behaviour around compaction, the inner LLM call goes through
`routeLLMByGlobalId(env, model)` which goes to the real D1 + real
provider — bypassing the test's `__llmOverrideScript`.

**Fix:** wire compaction's LLM call through the same `#routeLLM` indirection
that `#generate` uses for its main call. Pass
`deps: { llm: (env, id) => this.#routeLLM(id) }`. Then the same
`__setLLMOverride` script covers compaction too.

### 1.6 Title generation consumes a script turn from `__setLLMOverride` (**P2**)

**File:** `src/lib/server/durable_objects/ConversationDurableObject.ts:1485–1535`.

`#writeTitle` calls `this.#routeLLM(titleModel)` and chats. When tests set
an override with one scripted turn for the assistant message, the
title-gen call (kicked off via `ctx.waitUntil` from `#touchConversation`)
also pulls from the script. Result: either it consumes turn 2 (if the
test has multiple turns) or it gets `"FakeLLM: ran out of scripted
turns"` (which the title-gen catch handler swallows, falling back to
the truncated user message). Tests pass for the wrong reason; on a
slower CI run the order can race.

**Fix:** route title generation through a separate seam, e.g.
`__setTitleLLMOverride`, or have `#routeLLM` accept a "purpose" tag and
let tests script per-purpose.

### 1.7 `abortGeneration` loses meta snapshot (**P2**)

**File:** `src/lib/server/durable_objects/ConversationDurableObject.ts:574–595`.

When the user aborts mid-stream, the row is updated to
`status='complete'` with whatever content/parts existed. But
`started_at`, `first_token_at`, `last_chunk_json`, `usage_json`,
`provider` are *not* written. So the `MetaPanel` for an aborted message
shows nothing. The "complete" row is indistinguishable from one that
finished gracefully but without timing data.

**Fix:** in `abortGeneration`, write `started_at` / `first_token_at`
from the closure variables — which means hoisting them out of
`#generate` onto `#inProgress`, where `abortGeneration` can read them.

### 1.8 `appendDeltaPart` strips `textHtml` from the previous chunk (**P3**)

**File:** `src/lib/conversation-stream.ts:55–58`.

When a new delta lands and the trailing part is the same kind, the
helper preserves `textHtml` ("preserve existing textHtml so the UI
doesn't flicker to raw text"). Good. But the *next* delta then replaces
`text` to a longer string, leaving `textHtml` as the rendering of an
older revision until the streaming-markdown runner catches up. The UI
shows "old HTML" briefly while text grows — the same flicker the
comment claims to prevent, just in the other direction. Effectively
fine; document the trade-off.

### 1.10 Theme cache invalidation is only local to one isolate (**P3**)

**File:** `src/hooks.server.ts:25–41`,
`src/lib/settings.remote.ts:62`.

`themeCache` is a module-scope `let`. After the user saves a new theme,
`invalidateThemeCache()` clears the cache *in the isolate that handled
the save*. A user with two tabs on different isolates (smart placement
mostly avoids this for one user, but it can happen on cold isolates)
sees the old theme on the second tab for up to 30 s.

For single-user, fine. Document in a comment so the next person
debugging "why is my theme stale on tab 2?" has a hint.

### 1.11 `addPresetProvider` doesn't pre-check id collisions (**P3**)

**File:** `src/lib/providers.remote.ts:178–235`.

### 1.12 Sandbox SSH key cache misses key rotation (**P3**)

**File:** `src/lib/server/tools/sandbox.ts:18–52`.

`sshKeyInjected = new Set<string>()` keyed by `conversationId`.
Rotating `SANDBOX_SSH_KEY` doesn't bust the cache; conversations that
already had the old key remain stuck with it until the isolate cycles
or the conversation is destroyed.

**Fix:** key on `${conversationId}:${hashKey(key)}` so a key change
causes a re-injection on the next call.

Round-1 audit flagged this; not yet fixed.

### 1.13 `+page.server.ts` `withRenderedMarkdown` has a confused `??` chain (**P3**)

**File:** `src/routes/c/[id]/+page.server.ts:43–48`.

```ts
const contentHtml =
  typeof m.contentHtml === 'string' && m.contentHtml.length > 0
    ? m.contentHtml
    : m.content
      ? await renderMarkdown(m.content)
      : m.contentHtml ?? null;
```

When `contentHtml` is null/empty *and* `content` is empty, returns
`m.contentHtml ?? null` — which by definition was already null/empty,
so this branch always returns `null`. Simplify:

```ts
const contentHtml =
  m.contentHtml && m.contentHtml.length > 0
    ? m.contentHtml
    : m.content
      ? await renderMarkdown(m.content)
      : null;
```

### 1.14 `OpenAILLM`'s `flattenToText` drops non-text user content (**P3**)

**File:** `src/lib/server/llm/OpenAILLM.ts:241–248`.

`user` and `system` messages flatten to text only. Image/file blocks
defined in `LLM.ts:13–14` silently disappear. Anthropic adapter
faithfully maps images via `blocksToAnthropic`. So a multi-modal user
message routed to OpenAI-compat goes through with no images.

Either widen the OpenAI mapping to surface images via the
`{role:'user', content: [{type:'image_url', ...}]}` shape, or assert
loudly when a non-text block reaches `flattenToText` so the call site
knows to route differently.

Round-1 audit flagged this; not yet fixed.

## 2. Per-area notes

### 2.1 Durable Object (`src/lib/server/durable_objects/ConversationDurableObject.ts`)

The round-1 work cleaned up most of the cliff-edges (schema versioning,
MAX_TOOL_ITERATIONS boundary, tool-call atomicity, abort handling,
resume after eviction). The remaining items:

- **§1.3** — provider stream not cancelled on abort.
- **§1.5** — compaction LLM not routed through the test seam.
- **§1.6** — title generation shares the test-override LLM script.
- **§1.7** — abortGeneration loses meta snapshot.
- D1 fan-out on every `#generate` (§3 below).
- The `inflight` mirror's parts are mutated in-place after seeded by
  the live array (`appendText` / `appendThinking` mutate `last.text`).
  The DO test suite covers the happy path; the tool-cancel path is
  covered by tests added in round 1. Looks robust.
- Schema migrations: `_meta`-versioned, append-only, idempotent.
  Adding a new migration is one step now. Good.
- Resume tests cover happy path, multi-streaming-row dedupe, missing
  conversation_id, idempotent repeat-pokes, and synthetic error
  results for stranded streaming `tool_result`s. Solid.

### 2.2 LLM adapters (`src/lib/server/llm/`)

- **§1.4** — `routeLLM` instantiates a fresh SDK client per call.
- `Anthropic`'s `thinking` vs `reasoning.max_tokens` precedence is now
  explicit (`AnthropicLLM.ts:52–60`); the DO picks one in
  `ConversationDurableObject.ts:879` based on `providerType`. Good.
- `OpenAILLM`'s `partialToolCalls` finalisation is now hoisted into
  `finalizeToolCalls`, gated only by `emittedDone` for the `done`
  yield. No more double-emission. Good.
- `OpenAILLM` swallows post-`done` errors silently
  (`OpenAILLM.ts:147–149`). For a thrown SDK error after the stream
  completed, that's probably right. But "thrown after `done`" is also
  the symptom of a real bug elsewhere — it's worth a `console.warn`
  rather than total silence.
- `compactHistory` correctly subtracts `cacheReadInputTokens` from the
  budget estimate (line 83). The fallback drop-index logic is now
  bounded to `messages.length - minKeep`. The `wasCompacted` boolean is
  honest about whether anything was dropped.
- `OpenAILLM` `extra_content.google.thought_signature` round-trip is
  well-commented — sturdy little workaround.
- Both adapters share the same "iterate `partialToolCalls`,
  `JSON.parse(args)` with `_raw` fallback, yield `tool_call`" idiom.
  Hoist into a shared helper if a third adapter shows up; not worth a
  refactor with two.

### 2.3 Tools (`src/lib/server/tools/`)

- **§1.3 / §1.7 / §0.9** — abort signal not threaded from the DO into
  `registry.execute(...)`'s context.
- **§0.10** — `agent.ts` has belt-and-braces allowed-tools enforcement.
- `agent.ts` now propagates `result.citations` and `result.artifacts`
  out to the parent loop (lines 219–252). Round 1 fix held.
- `fetch_url.ts` streams the body to the cap with `getReader()` and
  cancels at the cap (lines 130–175). OOM safety. Good.
- `web_search.ts` clamps `count` to `[1, 10]` defensively (line 34).
- `ynab.ts` is still ~750 lines of similar-shaped tool factories.
  The round-1 audit deferred a DRY refactor; nothing has changed, and
  the shape is the same. If you adopt a tool factory pattern later
  (e.g. `ynabTool({name, description, inputSchema, exec})`), this is
  the natural sketch:
  ```ts
  function ynabTool<I, R>(meta, exec): Tool {
    return {
      definition: { name: meta.name, description: meta.description, inputSchema: meta.inputSchema },
      async execute(_ctx, input) {
        const args = (input ?? {}) as Partial<I>;
        for (const k of meta.required) if (!args[k]) return err(`Missing required parameter: ${k}`, 'invalid_input');
        const api = makeApi(meta.token);
        const result = await call(() => exec(api, args as I));
        if (isError(result)) return err(result.__error);
        return ok(meta.shape ? meta.shape(result) : result);
      },
    };
  }
  ```
  Each tool collapses to ~10 lines.
- `ynab.ts` constructs `new ynab.API(token)` per execute call
  (`makeApi`). For a single user this is fine, but ~150 ms of
  reconstruction overhead per call. Hoist into the `createYnabTools`
  factory closure.
- `ynab.ts` `updateTransactionTool` correctly skips the GET when all
  PUT-required fields are present (lines 641–691).
- `sandbox.ts` SSH key cache: §1.12.
- `registry.ts` returns a discriminated `errorCode` on failures
  (`'not_found'` / `'execution_failure'` / `'invalid_input'`). Tools
  use it. Good.

### 2.5 Streaming markdown (`src/lib/streaming-markdown.ts`)

- Cache pruning lives at lines 41–46 — drops keys whose `messageId` is
  no longer in `state.messages`. Round 1 fix held.
- The `inFlight` set + `requestAnimationFrame` schedule prevents
  redundant rendering while a delta is mid-flight. Solid.
- Concurrent renders of the same key are gated by `inFlight.has(key)`,
  but if a render for revision N is in flight when a delta lands and
  bumps to revision N+1, the in-flight render eventually writes its

### 2.6 Providers / D1 (`src/lib/server/providers/*`)

- **§1.4** — SDK client cache regression.
- **§0.4** — `migrate.ts:migrateLegacyModelList` is dead code.
- **§1.11** — `addPresetProvider` doesn't pre-check id collisions.
- `providers.remote.ts:62, 112, 162` use `await import('$lib/...')` for
  `getProvider`, `getModel`, `listModelsForProvider`. There is no
  obvious circular-import dependency that requires this. Top-level
  imports would be one less moving part.
- `provider_models.id` is non-unique within the provider (the primary
  key is `(provider_id, id)`), so two models with the same id under
  different providers is supported. The settings UI sorts/filters by
  `provider_id` + `sortOrder` correctly.
- Sort order is preserved on swap with two updates, which means the
  pair is briefly in an inconsistent state. For single-user this is
  fine. Wrapping in a D1 transaction would be defensive but not

## 3. Performance (rough order, highest payoff first)

1. **Cache the per-DO ConversationContext snapshot**
   (system_prompt, user_bio, thinking_budget, sub_agents,
   listAllModels, mcp_servers). 8–10 D1 round trips per turn → 1 (or 0
   on warm DO). The lookup covers what `#generate` and
   `#buildToolRegistry` need; invalidate on the relevant settings
   form actions or use a 30-second TTL like the theme cache. **§0.6.**
2. **Reuse SDK clients** in `routeLLM`. Drops connection setup per
   chat / compaction / title turn. **§1.4.**
3. **Forward AbortSignal** from the DO into `llm.chat()` and the
   tool-execution context. Reduces wasted provider tokens after abort.
   **§1.3 / §0.9.**
4. **Drop `messages.parts` once `parts_html` is universal.** Halves the
   per-row JSON. One-shot DO migration. **§0.8.**
5. **Cache parsed JSON shapes in DO memory while warm.** `#readMessages`
   does `JSON.parse` on every call for `parts`, `parts_html`,
   `tool_calls`, `tool_results`, `usage_json`, `last_chunk_json`. A
   small per-DO cache keyed by `(messageId, version)` would amortise.
6. **`ynab.ts` API client per call** — hoist into the factory. Single
   user so the impact is minimal, but it's free.
7. **Bootstrap CSS bundle** — `app.scss` imports the entire Bootstrap
   bundle. The app uses utility classes plus a few components. A
   tree-shaken Bootstrap entry would shrink the worker by hundreds of
   KB. (Round-1 audit flagged; deferred as cosmetic.)
8. **Shiki + KaTeX bundle** — the worker now ships every Shiki language
   in the curated list (`markdown.ts:10–25`). Worth running
   `wrangler deploy --dry-run` and checking the report; if it's near
   the 5 MiB cap, lazy-loading languages is a project.

## 4. Architecture

### 4.1 Triple-storage continues

`messages` row writes:
- `content`, `content_html` (text + rendered HTML)
- `thinking`, `thinking_html`
- `parts`, `parts_html` (JSON timelines, raw and rendered)
- `tool_calls`, `tool_results` (legacy JSON columns, now redundant
  because the same data is in `parts`)
- `usage_json`, `last_chunk_json`, `generation_json`

The `tool_calls`/`tool_results` columns are read by `buildLegacyParts`
for rows that pre-date `parts`. Once the operator has confirmed all
live rows have `parts`, schedule a one-shot migration to drop those
columns. Same for `parts_html` vs `parts` — see §0.8.

### 4.2 Reactive-state proxy hot path

`+page.svelte` keeps `convState: ConversationState` as a `$state`
object. Every SSE delta produces a brand-new outer object via
`patchMessage` (`conversation-stream.ts:32–45`), then a brand-new
messages array, then a new patched message, with the trailing part
(only) mutated in place. That's `O(messages.length)` array allocation
per token. For a 200-message conversation streaming a long answer,
that's 200 × 5000 = 1M shallow-copy ops over a turn.

The mutate-trailing-part trick at `appendDeltaPart` cuts the *part*
allocation, but the message-array spread above it still allocates.
For a small frontend the cost is invisible. If a 10-line refactor to
mutate the trailing message-array entry in place is on the table, take
it.

### 4.3 Sub-agents in production

`agent.ts` is now well-shaped: `routeLLM` injectable, signal threaded,
citations/artifacts propagated. The remaining gap is observability —
sub-agent runs don't surface intermediate state to the UI (they emit
no `tool_call` events upstream from the inner loop). For single-user
that's probably fine, but if a sub-agent run takes 30 s the operator
sees nothing. Consider plumbing a "sub-tool started/finished" SSE
event for transparency.

### 4.4 D1 / DO ownership boundary

D1 owns: `conversations`, `settings`, `mcp_servers`, `sub_agents`,
`providers`, `provider_models`. DO owns: `messages`, `artifacts`,
`_meta`. Round 1 noted the DO calls D1 several times during
generation; that's the cache-it-once recommendation in §3.1.

The conversation `updated_at` is bumped from D1 *and* via the DO's
`#touchConversation`. Consolidating on one of those would simplify.

### 4.5 `csrf.trustedOrigins: []`

Correct under Cloudflare Access. Round-1 note retained.

### 4.6 Service worker is a stake-out

`src/service-worker.ts` registers, claims clients, has no fetch
handler. Just enough to make iOS treat the page as installable. The
file is intentionally minimal and well-commented. No action.
